v1.16 Barbarendörfer Berichte-Auswertung

🏴 Barbarian Village Former

Wird von der Angriffsberichte-Übersicht (screen=report&mode=attack) aus gestartet und liest die dort bereits angezeigte Berichte-Liste direkt aus. Verarbeitet nur die aktuell angezeigte Seite - bei mehreren Seiten mit Berichten bitte das Script auf jeder Seite erneut ausführen. Aus den enthaltenen Spähdaten ermittelt es Mauer- und Gebäudelevel der Barbarendörfer und berechnet passende Truppenkombinationen aus einer gewählten Dorf-Gruppe: erst die Mauer mit Rammen/Äxten brechen, danach das gewählte Gebäude in mehreren Schritten mit Katapulten herunterstufen (bis zu einem Mindestlevel, in wählbaren Level-Schritten pro Befehl). Läuft auch in der mobilen Tribal-Wars-App.

Das Ergebnis ist eine Liste fertiger Workbench-Befehle (Base64-codierte Truppenzahlen) zum Kopieren – kompatibel mit dem TW Format-Konverter dieser Seite. Es wird nichts automatisch abgeschickt.

Installation (Desktop)

Ziehe diesen Button in deine Lesezeichenleiste. Öffne danach im Spiel die Berichte-Übersicht (Angriffsberichte) und klicke auf das Lesezeichen, um das Script zu starten.

Installation (Handy: iPhone/Android)

Auf dem Handy lässt sich das Bookmarklet nicht per Drag & Drop in die Lesezeichen-/Favoritenleiste ziehen. Stattdessen legst du zuerst ein ganz normales Lesezeichen an und ersetzt dessen Adresse danach durch den folgenden Code: 

javascript:$.getScript('https://marlongsimo.github.io/DS-Scripts/scripts/barbarianVillageFormer.js');

iPhone (Safari):

  1. Diese Seite (oder eine beliebige andere) über das Teilen-Symbol als Lesezeichen speichern.
  2. Unten auf das Bücher-Symbol tippen → „Lesezeichen“ → „Bearbeiten“ → das eben erstellte Lesezeichen antippen.
  3. Das Adressfeld leeren und den oben kopierten Code einfügen. Als Namen z.B. „Barbarian Village Former“ eintragen und speichern.
  4. Damit es schnell erreichbar ist: das Lesezeichen in den Ordner „Favoriten“ verschieben – es erscheint dann in der Schnellzugriffsleiste beim Öffnen eines neuen Tabs.
  5. Im Spiel die Berichte-Übersicht (Angriffsberichte) öffnen, dort einen neuen Tab öffnen und das Lesezeichen aus den Favoriten antippen.

Android (Chrome):

  1. Über das Menü (⋮) → „Lesezeichen“ → „Hinzufügen“ ein neues Lesezeichen anlegen.
  2. Das Lesezeichen antippen und halten → „Bearbeiten“, die URL löschen und durch den oben kopierten Code ersetzen, Namen anpassen und speichern.
  3. Damit es in der Schnellleiste erscheint: in den Chrome-Einstellungen unter „Lesezeichen“ die Lesezeichenleiste aktivieren, oder das Lesezeichen zum Startbildschirm hinzufügen.
  4. Im Spiel die Berichte-Übersicht (Angriffsberichte) öffnen und dort das Lesezeichen aus der Leiste antippen.

Nutzung auf eigene Verantwortung. Das Script sendet nichts automatisch ab, sondern erzeugt nur eine kopierbare Befehlsliste. Zusätzlich lädt es beim Start eine externe Bibliothek (twSDK.js) von twscripts.dev nach – es besteht also eine Laufzeitabhängigkeit von einer Drittanbieter-Domain. Bitte vorab die Regeln deines Servers zu Skripten/Bookmarklets prüfen. Das Script liest zum Auswerten die Berichte-Liste, die gerade angezeigt wird – achte also selbst darauf, auf der allgemeinen Angriffsberichte-Liste zu sein und nicht in einem eigenen Berichte-Ordner (z.B. „Hochladen“, „Archiv“); im Panel wird nach jedem Lauf angezeigt, welche Seite tatsächlich verwendet wurde.
Vorgeschichte (Stand v1.16): in früheren Versionen (v1.3–v1.15) startete dieses Script vom Versammlungsplatz aus und lud die Berichte-Liste dafür per Netzwerk-Request nach. Dabei wurde live wiederholt bestätigt, dass echte Angriffsberichte beim Ausführen des Scripts verschwinden konnten, während manuelles Öffnen desselben Berichte-Typs sie nachweislich nie löscht. Ein Zwischenfix (Laden über ein unsichtbares iframe statt per AJAX) führte zu einem weiteren, unabhängigen Problem: die mobile App-WebView leitete dabei sichtbar zur Berichte-Seite weiter, ohne dass zwei verschiedene Sandbox-Fixes das beheben konnten.

Lösung in v1.16: das Script läuft jetzt wieder von der Angriffsberichte-Übersicht aus und liest die Berichte- Liste direkt aus der bereits geladenen Seite – dafür ist überhaupt kein Netzwerk-Request mehr nötig. Damit entfällt für die Liste jedes Restrisiko (weder Datenverlust noch die Weiterleitung können hier noch auftreten, da schlicht keine Anfrage mehr stattfindet). Einzige Einschränkung: es wird nur die gerade angezeigte Seite verarbeitet, kein automatisches Weiterblättern mehr über mehrere Berichte-Seiten.

Verbleibende, noch nicht bestätigte Unsicherheit: einzelne Berichte werden weiterhin per Netzwerk abgerufen (über dasselbe iframe-Verfahren wie zuvor, da das die beste bekannte Verteidigung gegen den bestätigten Datenverlust ist). Es ist nicht bestätigt, ob dabei dieselbe Weiterleitungs-Problematik erneut auftreten kann – alle bisherigen Beobachtungen dazu stammen aus Läufen, bei denen zusätzlich die Liste per iframe geladen wurde. Bitte weiterhin testen und zurückmelden, ob (a) Berichte erhalten bleiben und (b) beim Öffnen einzelner Berichte eine Weiterleitung auftritt.
